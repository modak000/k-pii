# 실세계 성능 분석 — 정규식별 솔직 점검

본 문서는 k-pii 의 *각 정규식 패턴* 이 실제 한국 공공 문서에서 어떤 PII 를
잡고 못 잡는지 솔직히 정리한다. 70+ 케이스 실측 결과.

## 종합 결과

| 분류 | 건수 | 비고 |
|---|---|---|
| ✅ TP (제대로 검출) | 50+ | 의도된 catch 모두 성공 |
| ✅ TN (제대로 거부) | 8+ | FAKE 케이스·미할당 우편번호·체크섬 실패 카드 등 |
| ⚠️ 의도된 미검출 | 6 | 정책상 거부 (외래어·체크섬 실패·anchor 없음) |
| ⚠️ 진짜 FN | 2 | 공백 RRN (fix 적용), 외교 차량 (드문 포맷) |
| ❌ FP | 0 | 모든 거부 케이스 정상 거부 |

**실측 정확도:** 정형 한국 공공 문서에서 **거의 완벽**. 자유 자연어 인명만
룰 한계 (F1 ~0.33).

## 정규식별 상세 분석

### 1. RRN (주민등록번호) — `patterns/rrn.py`

```python
_PATTERN = re.compile(
    r"(?<![0-9])"
    r"([0-9]{6})"
    r"[-\s]?"           # 하이픈 / 공백 / 없음 (이 커밋에서 공백 추가)
    r"([0-9]{7})"
    r"(?![0-9])"
)
```

**잡는 것:**
- `880101-1234568` — 표준
- `8801011234568` — 하이픈 없음
- `880101 1234568` — 공백 (이번 fix)
- 7th digit 1·2·3·4·9·0 (한국인) — 외국인은 별도 FRN
- 후-2020 무작위화 RRN (체크섬 실패 → 신뢰도 0.7 유지)

**검증:**
- 날짜 유효성 (월 1-12, 일 valid, 윤년)
- 체크섬 (가중합 mod 11) — 통과 시 신뢰도 1.0
- 외국인 gender (5·6·7·8) → 자동으로 FRN 으로 분리

**못 잡는 것:**
- `880101/1234568` (슬래시) — 매우 드물어 의도적 미지원
- 14자리 이상 연속 숫자 안에 포함된 RRN — 다른 식별번호와 충돌 위험으로 거부

### 2. PHONE (전화번호) — `patterns/phone.py`

```python
_MOBILE = re.compile(
    r"(?<![0-9+])"
    r"(01[01679])"            # 010/011/016/017/018/019
    r"[-.\s]?"
    r"(\d{3,4})"
    r"[-.\s]?"
    r"(\d{4})"
    r"(?![0-9])"
)
```

**잡는 것 (모든 형태):**
- `010-1234-5678` `010.1234.5678` `010 1234 5678` `01012345678`
- `+82-10-1234-5678` `0082 10 1234 5678` (국제)
- 서울 `02-XXX-XXXX`, 지역 `031~064-XXX-XXXX`
- VoIP `070-XXXX-XXXX`
- 011/016/017/018/019 (구형 휴대전화)

**못 잡는 것:**
- 100% 일치하지 않는 *변형 표기* (예: `010-1234·5678` 의 ·)
- 일부 기업 ARS 번호 (1588-XXXX 등 — 별도 분류 필요)

### 3. EMAIL — `patterns/email.py`

```python
r"([A-Za-z0-9._%+\-]+@[A-Za-z0-9](?:[A-Za-z0-9\-]*[A-Za-z0-9])?(?:\.[A-Za-z0-9](?:[A-Za-z0-9\-]*[A-Za-z0-9])?)+)"
```

**잡는 것 (RFC 5322 실용 부분집합):**
- 표준: `user@example.com`
- 공공: `minsu@gov.go.kr`
- `+` filter: `user+tag@gmail.com`
- 점 포함: `first.last@company.co.kr`
- 학교: `student@snu.ac.kr`

**못 잡는 것:**
- 한글 로컬 (`한글@example.com`) — RFC 비표준, *실제로도 거의 안 씀*
- 따옴표 포함 ("user name"@example.com) — RFC 허용이나 실무 X

### 4. CARD (신용카드) — `patterns/card.py`

```python
# Luhn mod-10 검증 + IIN 첫 자리 화이트리스트 (2/3/4/5/6/9)
```

**잡는 것:**
- Visa (4), Mastercard (5), MC 신규 (2), AMEX/JCB (3), Discover/UnionPay (6), 한국 국내전용 (9)
- 하이픈/공백/없음 모두

**의도적 거부 (정탐):**
- BIN 0/1/7/8 시작 — 현행 카드 브랜드 미할당
- Luhn 체크섬 실패 — 진짜 카드 아닐 가능성 매우 큼
- `1111-2222-3333-4444` 같은 trivial Luhn 통과 → BIN 1 거부

### 5. BUSINESS_REG (사업자번호) — `patterns/business_reg.py`

```python
# 10자리 + 국세청 가중합 체크섬
```

**잡는 것 (체크섬 통과):**
- `120-81-47521` (카카오 예시)
- `1208147521` (하이픈 없음)

**의도적 거부:**
- 체크섬 실패한 임의 10자리 숫자 — 진짜 사업자번호 아님

### 6. ADDRESS (주소) — `patterns/address.py`

도로명 + 지번 두 형식.

**잡는 것:**
- 도로명: `서울특별시 종로구 세종대로 209`
- 시·도 + 시·군·구 (2단계까지): `경기도 성남시 분당구 정자로 1`
- 지번: `서울특별시 강남구 역삼동 123`

**Anchor 필수:**
- 시·도/시·군·구 토큰 *또는* "주소" 키워드 ≤20자 — 자연어 `테헤란로 152 인근` 같은 명사구는 거부

### 7. IP — `patterns/ip.py`

**잡는 것:**
- IPv4 표준 (옥텟 0-255 검증)
- IPv6 전체 / `::1` 단축 / `::ffff:1.2.3.4` IPv4-mapped

**의도적 거부:**
- `256.256.256.256` (잘못된 옥텟)

### 8. VEHICLE (차량번호) — `patterns/vehicle.py`

```python
# 50개 한글 용도 코드 화이트리스트
# 자가용 32자 + 영업용 4자 + 렌터카 3자 + 외교 5자 + 군용 5자 + 배달 1자
```

**잡는 것:**
- 자가용 `12가3456`, `123자4567`
- 영업용 (택시·택배) `87바1234`
- 렌터카 `99하1234`
- 군용 `01국0001`

**의도적 거부:**
- `30강 1234` — `강` 은 차량 용도 코드 아님 (자연어 한글)
- `12가0000` — 뒷 4자리 0000 placeholder

### 9. PASSPORT — `patterns/passport.py`

**잡는 것 (14 prefix):**
- 단일: `M/S/G/O/D/R/T` (general/single/general/official/diplomatic/resident/travel)
- 두 글자 (2024.12.16~): `PP/PM/PS/PO/PD/PR/PT`

**의도적 거부:**
- `A12345678` — A 는 한국 여권 prefix 아님

### 10. MEDICAL_INSURANCE — `patterns/medical_insurance.py`

**잡는 것:**
- "건강보험"/"의료보험"/"보험증" 키워드 ≤25자 앞 + 11자리

**의도적 거부:**
- 11자리 단독 — 휴대전화 (010+8자리) 와 충돌

### 11. PERSON — `patterns/person.py` (룰 + 점수)

가장 복잡한 검출기. 18+ 신호 점수 시스템.

**잡는 것:**
- 정형 양식: "성명: 홍길동" "신청인: 김민수" → ✅✅✅
- 직책 인접: "기획재정부 김민수 과장" → ✅✅
- 자유 자연어 (조사 인접): "홍길동이 발표했다" → ✅
- 한국 인명 음절 통계 통과: "장혁" "이범수" 등 → ✅
- 같은 문장 인명 나열: "김지우와 이범수가" → 상호 보강

**의도적 거부:**
- 외국인명 (왕가위·소피마르소) — 가명화 대상 아님 (정책)
- 부분 가명 (홍OO·박씨·A씨) — 이미 가명화된 표기
- 일반 어휘 (검토·이날·자신 등) — 400+ common_words

**못 잡는 것 (룰 한계):**
- 한자 인명 (洪吉童) — 별도 hanja_to_hangul 모듈 있으나 검출엔 미통합
- 매우 특이한 이름 패턴 (음절 통계 미반영)
- 가명화된 외부 문서의 마스킹 패턴

### 12. KCD / 처방번호 / EDI 약품 / 사건번호 / PNU

- **KCD**: 알파벳 1자 + 숫자 2자 + (.숫자) — 소수점 형태 또는 키워드 anchor 필수
- **PRESCRIPTION_ID**: 12자리 = YYYYMMDD + 4자리, *키워드 anchor 필수*
- **EDI_DRUG**: 9자리 또는 13자리, 한국 국가코드 (880/881) + 키워드 필수
- **COURT_CASE**: `YYYY` + 50개 부호문자 화이트리스트 + 일련번호
- **PNU**: 19자리, 시도 코드 (11~50) + 본번 0 거부

## 정형 vs 자연어 문서별 예상 정확도

| 문서 유형 | 예상 F1 | 이유 |
|---|---|---|
| 정형 공문서 (결재·민원·인사 양식) | **0.95-1.0** | "성명: ", "주민번호: " 같은 라벨 활용 |
| 표 형식 (CSV·XLSX) — 컬럼 헤더 매핑 | **1.0** | 컬럼 라벨 명시적, 100% 카테고리 단언 |
| 신문기사·뉴스 자연어 | **0.3-0.4** | 룰 기반 한계 (KLUE-NER 측정) |
| 자유 이메일·메신저 텍스트 | **0.4-0.6** | 컨텍스트 약함 |
| OCR 출력 (스캔 문서) | **0.5-0.7** | 오인식 보정 X (별도 작업 필요) |

## 권장 사용처

✅ **즉시 production:**
- 공공기관 결재 시스템 - 1차 자동 가명화 + 사람 검토
- 민원 시스템 - 응대문 출력 전 PII 차단
- HR 데이터 분석 - 컬럼 단위 가명화
- 데이터 분석 전 정제 (RRN/카드/계좌 자동 차단)

⚠️ **검토 큐와 함께:**
- 자유 자연어 SNS·이메일 본문
- 회의록·인터뷰 텍스트

❌ **단독 사용 비권장:**
- 의료 차트 자유 기록 (한자·약자 많음)
- 외교 문서 (외국인명 핵심)
- 학술 논문 (인명·기관 빈출하나 PII 분리 미묘)

## 솔직한 결론

본 라이브러리는 **결정적 PII (체크섬 검증 가능한 카테고리)** 와 **정형 한국
공공 문서** 에서 ML 보다 *더 정확*. 자유 자연어 인명은 룰 한계가 있어
사람 검토 워크플로우 (`k_pii.review`) 또는 **OpenAI Privacy Filter
hybrid** 와 함께 사용 권장.
