# 합성 평가 코퍼스 샘플

> `k_pii.eval.synth` 의 6 템플릿 각 1건씩 — 실제로 어떤 문서가 생성되어
> 회귀 감지 sanity check 에 쓰이는지 확인용. **이 코퍼스의 F1 = 1.000 은
> 실제 정확도가 아닌 회귀 감지 점수**. 실제 정확도는 KDPII 53,778 문서
> 기준 micro F1 = 0.650 (`docs/real_data_benchmark.md` 참조).

## 왜 합성 F1 = 1.000 인가

각 템플릿이 **명확한 필드 라벨 (`성명:`, `주민등록번호:`, `주소:`)** 으로
모든 PII 를 anchor 한다. KDPII 같은 대화체 ("어릴 때 서울 동대문구에서 커서")
와 달리, 합성은 anchor 가 100% 명확해서 검출이 매우 쉽다. 그래서 1.000 도
가능하지만, *production 실데이터* 에서는 그렇지 않음.

용도: **새 룰/검출기 추가 시 합성 점수가 0.95 미만으로 떨어지면 회귀 신호.**

## 6 템플릿 샘플

### 1. meeting_minutes (회의록)

```
[회의록]

참석자: 송지효 팀장, 남궁민수 주무관

논의 사항:
1. 청사 출입 차량 87나4321의 통행 허용 여부.
2. 다음 주 보고회 일정 협의.
```

Gold 라벨: PERSON × 2, VEHICLE × 1

### 2. civil_petition (민원 답변서)

```
[민원 답변서]

민원인: 김민수
주민등록번호: 950101-2345676
연락처: 010-1234-5678
주소: 서울특별시 강남구 역삼동 123

귀하의 민원에 다음과 같이 답변드립니다.
김민수 님께서 제출하신 사항은 관련 부서에서 검토 중이며,
처리 결과는 통보드릴 예정입니다.
```

Gold 라벨: PERSON × 2 (반복), RRN, PHONE, ADDRESS

**가명화 결과 (STRICT + tokenize):**

```
[민원 답변서]

민원인: <PERSON_1>
주민등록번호: <RRN_1>
연락처: <PHONE_1>
주소: <ADDRESS_1>

귀하의 민원에 다음과 같이 답변드립니다.
<PERSON_1> 님께서 제출하신 사항은 관련 부서에서 검토 중이며,
처리 결과는 통보드릴 예정입니다.
```

→ 같은 김민수가 2번 등장 → 같은 토큰 `<PERSON_1>` 으로 일관성 유지.

## 실데이터와의 차이

| 측면 | 합성 (회귀 감지) | KDPII (실데이터) |
|---|---|---|
| 도메인 | 한국 공문서 양식 (좁음) | 한국어 일상 대화 (넓음) |
| anchor | 명확한 필드 라벨 | 대부분 모호 ("어릴 때 서울...") |
| 문서 수 | 60 (각 seed) | 53,778 |
| 라벨 종류 | 7 (PERSON/RRN/PHONE/EMAIL/ADDRESS/VEHICLE/DOC_ID) | 22 |
| F1 | 1.000 (과적합 상한) | 0.650 (정직한 값) |

## 인터랙티브 시각화

`docs/synthetic_sample.html` — civil_petition 1건의 가명화 결과 + 라벨별
색상 코딩 + 결합 위험도 자동 평가.

## 재현

```bash
python -m k_pii.eval.benchmark -n 60 --seed 0
# 라벨별 P/R/F1 출력 (모두 1.000 이 정상 — 회귀 시 0.95 미만)
```
